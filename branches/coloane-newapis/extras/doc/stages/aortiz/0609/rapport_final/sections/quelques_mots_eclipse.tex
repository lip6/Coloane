\section{Quelques mots sur Eclipse}
\subsection{Tout est projet}
\par
Un aspect d'Eclipse qui est, au départ, assez déroutant, est
sa gestion des ``projets''.\\
Avant de faire ce stage, j'avais tendance à voir un projet
comme une entité séparée en structures plus petites
pour les différentes phases du projet (modélisation,
développement, tests, déploiement, ...).
\par
Avec Eclipse, ce sont ces différentes structures qui sont
des projets (par exemple un projet pour l'écriture de tout
ou partie du plug-un, un pour les tests, un pour
le déploiement, ...).
\par
Il faut bien avoir cette représentation en tête pour comprendre
(un peu) Eclipse, et ne pas croire qu'un projet
au sens Eclipse du terme va comprendre tout le processus
du début à la fin.


\subsection{Un plug-in n'est pas un plug-in}

\subsubsection{Cas de Subclipse}
\par
Un plug-in, comme son nom l'indique en partie, est simplement
un bout de code (auquel sont souvent associés d'autres fichiers
comme les icônes, ...) qui se ``branche'' sur une
architecture déjà en place.\\
Par exemple, si on veut explorer un dépôt SVN à
travers Eclipse, il suffit d'installer le plug-in
Subclipse.\\
Après installation de ce plug-in \textit{via} le site
d'update situé à l'adresse http://subclipse.tigris.org/update\_1.0.x,
on s'apperçoit, en regardant dans Help->About Eclipse SDK->
Plug-in Details que ce n'est pas un plug-in qui a été
installé, mais trois :
\begin{itemize}
\item Subclipse
\item SVN Team Provider Core
\item SVN Team Provider UI
\end{itemize}
\par
Par contre, si on regarde dans les Feature Details,
il n'y a qu'une feature Subclipse qui a été installée.

\subsubsection{Explications}
\par
Eclipse n'installe pas, \textit{via} son outil d'installation
et de mises-à-jour, des plug-ins mais des features (fonctionnalités).\\
Qu'est-ce qu'une feature ? Un groupement de plug-ins.\\
Quand on veut ajouter une fonctionnalité à Eclipse, il est parfois
nécessaire, pour des raisons de maintenabilité,
de mettre différentes parties de cette fonctionnalité
(par exemple la GUI et le code fonctionnel du plug-in) dans plusieurs
plug-ins.\\
Dans ce cas, il serait absurde de pouvoir installer les plug-ins
de manière indépendante (par exemple installer seulement
la partie interface graphique).\\
C'est pour cela qu'Eclipse se limite à l'installation de features,
même si rien n'empêche un utilisateur d'installer un plug-in
tout seul en le copiant simplement dans le répertoire plugins
d'Eclipse (mais là Eclipse n'y est pour rien si l'utilisateur
fait quelque chose d'absurde).


\subsection{Eclipse et les insectes}
\par
La taille du projet Eclipse implique qu'il est difficile de
le garantir 100\% sans bugs.\\
Lors de mon stage, j'ai été confronté à des ``événements
inexplicables'', qui ont été résolus en redémarrant simplement
Eclipse.\\
Par exemple, Eclipse refusait de voir un projet comme un plug-in
et après un redémarrage le projet était de nouveau vu comme
un plug-in.
\par
Les autres difficultés que j'ai rencontrées, qui semblaient
inexplicables au début, ont été solutionnées (elles seront
détaillées dans la suite).

\subsection{Prérequis}
\par
Le lecteur doit au moins savoir comment créer un projet et comment
installer un nouveau plug-in en utilisant le gestionnaire
d'installation d'Eclipse.
Dans la suite de ce document, je considérerai que l'automatic build
est désactivé (``Project -> Build Automatically'').\\
Les manipulations qui sont indiquées se font dans la perspective
de PDE, sauf mention contraire.
